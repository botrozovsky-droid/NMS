## 🎯 Резюме

**OpenClaw Memory System** — это система долговременной памяти для AI агентов, построенная на принципах работы человеческого мозга.

Система позволяет AI:
- **Помнить прошлые взаимодействия** недели и месяцы спустя
- **Учиться на опыте** и не повторять ошибки
- **Выявлять паттерны** в действиях пользователя
- **Забывать неважное** и фокусироваться на главном
- **Самостоятельно оптимизироваться** без ручной настройки

**Ключевая особенность:** Система работает полностью локально и не требует затрат на облачные API.

## 🤔 Проблема

### Современные AI агенты страдают "амнезией"

**Типичный сценарий:**

```
День 1:
Пользователь: "Я предпочитаю Python, а не JavaScript"
AI: "Хорошо, запомнил!"

День 7:
Пользователь: "Создай проект"
AI: "На каком языке? JavaScript или Python?"
Пользователь: 😤 "Я же говорил - Python!"
```

### Почему это происходит?

**1. Ограниченный контекст**
- Большинство AI моделей "видят" только последние 10-50 сообщений
- Старые разговоры полностью забываются

**2. Отсутствие долговременной памяти**
- Текущие решения: простой лог или база данных
- Проблема: всё хранится одинаково (важное и неважжное)
- Результат: информационный шум

**3. Нет естественного забывания**
- Устаревшая информация засоряет память
- Невозможно отличить актуальное от устаревшего

**4. Отсутствие связей между фактами**
- "Пользователь любит Python" сохранено
- "Пользователь создал проект на FastAPI" сохранено
- Но AI не понимает что FastAPI → Python → предпочтение пользователя

## 💡 Решение

### Система памяти, работающая как человеческий мозг

Мы создали **4-слойную систему**, имитирующую работу биологической памяти:

```
┌─────────────────────────────────────┐
│ 1. Сенсорный Буфер                  │
│    "Что происходит прямо сейчас"   │
│    Время жизни: 1 разговор         │
└─────────────────────────────────────┘
              ↓
┌─────────────────────────────────────┐
│ 2. Гиппокамп (кратковременная)      │
│    "Что произошло сегодня"         │
│    Время жизни: до ночи            │
└─────────────────────────────────────┘
              ↓ (каждую ночь)
┌─────────────────────────────────────┐
│ 3. Неокортекс (долговременная)      │
│    "Что я знаю вообще"             │
│    Время жизни: ~30 дней           │
└─────────────────────────────────────┘
              ↓
┌─────────────────────────────────────┐
│ 4. Процедурная память               │
│    "Как делать что-то"             │
│    Время жизни: постоянно          │
└─────────────────────────────────────┘
```

### Ключевые принципы

**1. Не всё одинаково важно**
- Ошибки запоминаются лучше (×5 приоритет)
- Код и файлы важнее обычного текста (×2)
- Мелкий talk забывается быстрее

**2. Частое усиливается**
- "Python упоминается 10 раз" → крепкая связь
- "Java упоминался 1 раз" → слабая связь → забудется

**3. Старое забывается**
- Информация 30-дневной давности ослабевает
- Если не используется → удаляется

**4. Связи важнее фактов**
- Не просто "Python" и "FastAPI"
- А "Python" ←связан→ "FastAPI" ←связан→ "веб-разработка"

## 🔧 Как это работает

### В течение дня (режим реального времени)

**1. Захват событий**
```
Пользователь пишет → AI отвечает → Событие сохраняется
                                    с оценкой важности
```

**Пример:**
```javascript
Событие: "Ошибка чтения файла config.json"
Важность: 5.0 (высокая, это ошибка!)
Сохранено в: Гиппокамп
```

**2. Индексирование**
```
День → Сессия → События
2026-04-10 → session-123 → [event1, event2, event3]
```

Быстрый поиск: "что было 10 апреля?"

---

### Ночью в 3:00 (консолидация)

**Происходит "переработка" информации:**

```
1. Система берёт важные события дня
   (например, 20 из 100)

2. AI модель (Gemini API) анализирует:
   "О чём эти события? Какие концепты?
    Как они связаны?"

3. Извлекает знания:
   Концепты: ["config.json", "ошибка", "чтение файла"]
   Связи: config.json ←связан→ ошибка чтения

4. Обновляет граф знаний:
   - Создаёт новые узлы (если нужно)
   - Усиливает существующие связи
   - Ослабляет старые, неиспользуемые

5. "Подрезает" слабые связи
   (вес < 0.01 → удалить)
```

**Результат:** Из 100 событий осталось 15 концептов с 30 связями между ними.

---

### Каждое воскресенье (самооптимизация)

**Система анализирует сама себя:**

```
1. Как эффективно используется память?
   - Слишком много узлов? → Увеличить "забывание"
   - Слишком мало? → Уменьшить порог важности

2. Насколько хорошо работает консолидация?
   - Много ошибок? → Снизить требования
   - Всё отлично? → Повысить требования

3. Автоматически меняет параметры:
   - Скорость обучения
   - Скорость забывания
   - Порог важности
```

**Результат:** Система сама подстраивается под ваши паттерны использования.

## 🏗️ Архитектура

### Слой 1: Сенсорный Буфер

**Назначение:** Захват всех событий в реальном времени

**Что сохраняется:**
- Сообщения пользователя
- Ответы AI
- Вызовы инструментов (чтение файлов, выполнение команд)
- Ошибки
- Изменения кода

**Время жизни:** Одна сессия разговора

**Формат:**
```json
{
  "type": "tool_call",
  "toolName": "Read",
  "file": "/config/app.json",
  "result": "success",
  "timestamp": 1775819766869
}
```

---

### Слой 2: Гиппокамп (Episodic Memory)

**Назначение:** Кратковременное хранение эпизодов

**Что здесь:**
- События сгруппированы по датам и сессиям
- Каждое событие имеет оценку важности (0-10)
- Индекс для быстрого поиска по дате

**Время жизни:** До ночной консолидации

**Структура:**
```
hippocampus/
├── sessions/
│   ├── 2026-04-10-session-123.json  ← События этой сессии
│   └── 2026-04-10-session-456.json
├── daily-index.json                 ← Индекс: дата → сессии
└── synaptic-candidates.json         ← Очередь на консолидацию
```

**Пример записи:**
```json
{
  "id": "event-789",
  "importance": 5.0,
  "event": {
    "type": "error",
    "message": "File not found: config.json"
  },
  "consolidated": false
}
```

---

### Слой 3: Неокортекс (Semantic Memory)

**Назначение:** Долговременное хранение знаний в виде графа

**Что здесь:**
- **Узлы** = концепты (Python, config.json, ошибка, FastAPI)
- **Рёбра** = связи между концептами с весами (0.0-1.0)
- **Веса** = насколько сильна связь

**Время жизни:** ~30 дней (зависит от использования)

**Пример графа:**
```
[Python] ←(0.9)→ [FastAPI] ←(0.7)→ [веб-разработка]
    ↓(0.8)
[предпочтение пользователя]
```

**Обновление весов:**
```
Связь использована → вес увеличивается (Hebbian learning)
Связь не используется → вес уменьшается (Temporal decay)
Вес < 0.01 → связь удаляется (Pruning)
```

**Формат:**
```json
{
  "nodes": {
    "python": {
      "id": "python",
      "type": "language",
      "weight": 0.85,
      "activations": 15,
      "lastActivation": 1775819766869
    }
  },
  "edges": {
    "python→fastapi": {
      "source": "python",
      "target": "fastapi",
      "weight": 0.9,
      "coActivations": 8
    }
  }
}
```

---

### Слой 4: Процедурная Память

**Назначение:** Паттерны действий и предпочтения

**Что здесь:**
- **Паттерны действий:** "Как делать X"
- **Предпочтения:** "Что любит пользователь"
- **Успешность:** Сколько раз сработало/не сработало

**Время жизни:** Постоянно

**Примеры:**

**Паттерн действия:**
```json
{
  "git_commit_flow": {
    "action": {
      "steps": ["git add", "git commit", "git push"]
    },
    "executions": 25,
    "successes": 23,
    "failures": 2,
    "weight": 0.92,
    "lastUsed": 1775819766869
  }
}
```

**Предпочтение:**
```json
{
  "preferred_language": {
    "value": "Python",
    "confidence": 0.95,
    "updateCount": 12,
    "lastUpdated": 1775819766869
  }
}
```

## 🧬 Научная основа

### 1. Hebbian Learning (1949)

**Принцип:** "Neurons that fire together, wire together"

**На практике:**
```
Концепты "Python" и "FastAPI" упоминаются вместе
→ Связь между ними усиливается
→ В будущем при упоминании Python, система вспомнит FastAPI
```

**Формула:**
```
new_weight = min(1.0, old_weight + learning_rate)
```

---

### 2. Forgetting Curve / Ebbinghaus (1885)

**Принцип:** Память ослабевает экспоненциально со временем

**График:**
```
Вес связи
   1.0 ┤●
       │  ●
   0.5 ┤    ●
       │      ●●
   0.0 ┤        ●●●●
       └──────────────> Время
       0   15   30 дней
```

**Формула:**
```
new_weight = old_weight × exp(-ln(2) × time / half_life)
```

**На практике:**
- Информация 30-дневной давности → вес × 0.5
- Информация 60-дневной давности → вес × 0.25
- И так далее

---

### 3. Hippocampal-Neocortical Consolidation (McClelland, 1995)

**Принцип:** Мозг переносит информацию из гиппокампа в неокортекс во время сна

**В системе:**
```
День: События → Гиппокамп (быстро, временно)
Ночь: Гиппокамп → Неокортекс (медленно, долговременно)
```

**Зачем:**
- Гиппокамп быстрый но ограниченный (как RAM)
- Неокортекс медленный но огромный (как SSD)
- Разделение позволяет оптимизировать оба процесса

---

### 4. Attention Mechanism

**Принцип:** Не вся информация одинаково важна

**В системе:**
```javascript
Важность события =
  (если ошибка) × 5.0 +
  (если код/файлы) × 2.0 +
  (если обычный текст) × 1.0
```

**Примеры:**
- Ошибка: `importance = 5.0`
- Изменение 3 файлов: `importance = 2.0`
- "Привет, как дела?": `importance = 1.0`

**Результат:** Система фокусируется на важном, игнорирует шум.

## ⚡ Преимущества

### 1. 💰 Минимальная стоимость эксплуатации

**Как это возможно?**
- Вся тяжёлая работа (консолидация) делается через Gemini API
- Эффективное использование API - только для консолидации
- Локальное хранение данных (без облачных БД)

**Сравнение с облачными решениями:**
```
Облачная база векторов (Pinecone):
- $0.096 за 100K запросов
- ~$30-50/месяц

Наша система (v0.3.3):
- $0.58/месяц (только Gemini API)

50-100x дешевле!
```

---

### 2. 🧠 Естественное поведение памяти

**Как у человека:**
- Важное помнится дольше
- Частое помнится лучше
- Старое забывается постепенно
- Связанные факты вспоминаются вместе

**Пример:**
```
Пользователь: "Помнишь тот баг с конфигом?"

Система думает:
1. Ищет "баг" → находит несколько
2. Ищет "конфиг" → находит config.json
3. Смотрит связи: config.json ←связан→ ошибка чтения
4. Находит: "Да, 2 недели назад была ошибка ENOENT"
```

---

### 3. 🔄 Самооптимизация

**Система учится сама себя настраивать:**

**Сценарий 1:** Граф слишком большой
```
Мета-обучение видит: 50,000 узлов, памяти мало
Решение: Увеличить скорость забывания
Результат: Слабые связи удаляются быстрее
```

**Сценарий 2:** Консолидация пропускает важное
```
Мета-обучение видит: Много событий не попадают в граф
Решение: Снизить порог важности
Результат: Больше событий консолидируются
```

---

### 4. 📈 Масштабируемость

**Автоматическая защита от переполнения:**

```
1. Temporal Decay → старое ослабевает
2. Pruning → слабое удаляется (вес < 0.01)
3. Consolidation → только важное сохраняется
```

**Результат:**
- 10,000 событий/день → ~500 узлов в графе
- Граф не растёт до бесконечности
- Остаётся только самое важное

---

### 5. 🔗 Контекстное понимание

**Система находит связи автоматически:**

**Пример:**
```
События:
1. "Создали FastAPI проект"
2. "Используем Python 3.11"
3. "Ошибка импорта в main.py"

Консолидация извлекает:
Узлы: FastAPI, Python, main.py, ошибка
Связи:
  FastAPI ←→ Python (оба упоминались)
  main.py ←→ ошибка (связаны по контексту)
  FastAPI ←→ main.py (файл проекта)
```

**Результат:** В будущем при вопросе "что с FastAPI?" система вспомнит про main.py и ошибку.

## 🎯 Применение

### Use Case 1: Персональный AI ассистент

**Проблема:**
```
Пользователь постоянно повторяет свои предпочтения:
- "Я работаю на Windows"
- "Используй Python, не JavaScript"
- "Мой сервер на 192.168.1.100"
```

**Решение:**
```
После 2-3 упоминаний → система запоминает
В будущем:
  "Создай проект" → автоматически Python на Windows
  "Проверь сервер" → автоматически 192.168.1.100
```

---

### Use Case 2: Долгосрочные проекты

**Проблема:**
```
Проект длится 3 месяца
AI забывает что было в начале
Приходится искать в истории чата
```

**Решение:**
```
Граф знаний накапливает информацию:
- Архитектурные решения
- Частые ошибки и их решения
- Структура проекта
- Зависимости между компонентами

Через 2 месяца AI всё ещё помнит почему выбрали FastAPI, а не Flask
```

---

### Use Case 3: Обучение на ошибках

**Проблема:**
```
AI предлагает решение → ошибка
Через неделю AI предлагает то же решение снова
```

**Решение:**
```
Ошибки сохраняются с высоким приоритетом (×5)
При следующем похожем вопросе:
  AI: "Помню, в прошлый раз это вызвало ошибку X,
       попробуем альтернативный подход"
```

---

### Use Case 4: Командная работа

**Проблема:**
```
Несколько человек работают с одним AI
AI не помнит соглашений команды
```

**Решение:**
```
Процедурная память хранит:
- Стандарты кода команды
- Naming conventions
- Workflow (git flow, code review)

Новый член команды спрашивает → AI объясняет "как принято здесь"
```
